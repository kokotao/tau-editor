# PLAN-001: V1 开发里程碑与执行计划

**文档版本：** 1.0  
**创建日期：** 2026-03-12  
**状态：** Draft  
**关联文档：** `PRD-001-轻量桌面文本编辑器正式产品需求文档.md`

---

## 1. 目标

本计划用于把 V1 产品需求拆解为可执行的开发里程碑，帮助团队按照“先稳定核心闭环，再增强效率能力”的顺序推进。

V1 的唯一总目标是：

把“空启动 -> 打开 -> 编辑 -> 保存 -> 恢复”做成稳定、可分发、可持续迭代的桌面编辑器基础版本。

---

## 2. 执行原则

- 先修正产品模型，再补体验细节
- 先保证文件内容与 Tab 关系稳定，再做效率功能
- 先保证桌面分发和恢复闭环，再做高级能力
- 每个里程碑都必须有可验证的验收出口

---

## 3. 里程碑总览

| 里程碑 | 名称 | 目标 | 预计输出 |
|------|------|------|---------|
| M1 | 产品骨架稳定化 | 建立产品模式、欢迎态、文件入口 | 空启动正确、单文件/工作区可区分 |
| M2 | 编辑模型重构 | 建立稳定 Tab/内容/保存模型 | 内容真源统一、保存链路稳定 |
| M3 | 恢复与保护闭环 | 建立会话恢复、自动保存、退出保护 | 用户不易丢内容 |
| M4 | 桌面发布可用化 | 完成多平台打包和 macOS 通用包 | 可发布安装包 |
| M5 | 高频体验增强 | 最近项目、快速入口、查找替换准备 | 接近可日用版本 |

---

## 4. 里程碑详细拆分

## M1: 产品骨架稳定化

### 4.1 目标

- 明确 `empty` / `single-file` / `workspace` 三种模式
- 修正空启动体验
- 让“打开文件”和“打开文件夹”成为真正的产品主入口

### 4.2 范围

- 欢迎态
- 空侧栏状态
- 工作区标题
- 打开文件
- 打开文件夹
- 最近项目/最近文件的数据结构预留

### 4.3 验收标准

- 未打开工作区时，不再显示假文件树
- 空启动界面在 5 秒内可理解
- 单文件模式不依赖工作区存在
- 打开文件夹后资源树只展示真实目录

### 4.4 依赖

- `workspaceStore`
- `dialogService`

---

## M2: 编辑模型重构

### 4.5 目标

- 统一 `tabsStore` 为内容真源
- 降低 `App.vue` 业务耦合
- 建立可预测的 Tab/保存/重命名模型

### 4.6 范围

- `tabsStore` 重构
- `editorStore` 职责收缩
- `fileService` / `tabService`
- Tab 重命名
- 新建文件可编辑
- 单编辑器实例稳定切换

### 4.7 验收标准

- Tab 切换不串内容
- 新建文件立即可编辑
- 未保存文件首次保存自动走另存为
- 已保存文件重命名后路径同步正确

### 4.8 依赖

- `workspaceStore`
- `fileService`
- `tabService`

---

## M3: 恢复与保护闭环

### 4.9 目标

- 建立用户内容保护机制
- 让“关闭、崩溃、重启”都具备可恢复性

### 4.10 范围

- `sessionService`
- 会话快照持久化
- 自动保存策略
- 退出确认统一入口
- 恢复未保存草稿

### 4.11 验收标准

- 关闭应用时存在未保存内容必须有确认
- 已命名文件可自动保存
- 未命名文件可恢复草稿
- 重启应用后能恢复上次标签和活动项

### 4.12 依赖

- `sessionService`
- `windowService`
- `settingsStore`

---

## M4: 桌面发布可用化

### 4.13 目标

- 完成稳定构建与产物输出
- 保证 macOS Intel/M 系列基础兼容

### 4.14 范围

- GitHub Actions 发布流程
- macOS universal target
- 产物目录统一
- 安装说明
- 失败重试与构建可观察性

### 4.15 验收标准

- GitHub Actions 能产出 macOS universal bundle
- Windows / Linux / macOS 产物路径一致可追踪
- 安装说明与产物一致

### 4.16 依赖

- Tauri 打包配置
- Release workflow

---

## M5: 高频体验增强

### 4.17 目标

- 让编辑器进入“能长期用”的阶段

### 4.18 范围

- 最近项目展示
- 最近文件展示
- 当前文件查找替换
- 命令面板预埋
- 快速打开预埋
- 外部文件变化监听预埋

### 4.19 验收标准

- 用户第二次打开应用可以快速回到上次上下文
- 查找替换具备基本可用性
- 命令系统能承接未来入口扩展

### 4.20 依赖

- `command registry`
- `sessionService`
- `workspaceStore`

## 7. 风险控制

### 风险 1：里程碑交叉污染

表现：

- UI、状态、发布、会话恢复同时改，回归面太大

控制策略：

- 每个里程碑提交尽量聚焦一个主目标

### 风险 2：状态模型边改边兼容

表现：

- 旧逻辑和新逻辑并存，容易继续串内容

控制策略：

- 明确 `tabsStore` 是唯一内容真源

### 风险 3：先做体验功能导致基础不稳

表现：

- 命令面板、查找替换先上，但保存恢复仍不稳

控制策略：

- M1-M3 未完成前，不进入复杂效率功能的大规模开发

---

## 8. 结论

对当前项目来说，最重要的不是“做更多功能”，而是按顺序做对功能。

推荐执行主线如下：

1. 建立正确的产品模式  
2. 建立稳定的编辑模型  
3. 建立可恢复的内容保护链路  
4. 建立可信的桌面发布能力  
5. 再叠加高频效率体验

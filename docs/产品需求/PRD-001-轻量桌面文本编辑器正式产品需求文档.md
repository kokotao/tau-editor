**适用范围：** `text-editor` 桌面端产品

---

## 1. 文档目标

本 PRD 用于统一当前编辑器产品的产品目标、范围边界、核心用户场景、功能优先级和验收标准，解决当前项目中“功能点存在，但整体产品感、信息架构与桌面体验不统一”的问题。

本 PRD 的目标不是继续堆功能，而是把产品明确为一个可以持续演进的桌面文本编辑器。

---

## 2. 产品定义

### 2.1 产品名称

轻量桌面文本编辑器

### 2.2 产品定位

一款面向中文用户的轻量级桌面文本/代码编辑器，强调以下能力：

- 快速启动，低学习成本
- 支持单文件编辑和工作区编辑两种模式
- 默认界面清爽、审美在线、中文友好
- 保留开发者常用能力，但不做重型 IDE
- 桌面体验优先，而不是简单把 Web 页面打包成应用

### 2.3 一句话价值主张

比记事本更专业，比 IDE 更轻盈的桌面编辑器。

---

## 3. 目标用户

### 3.1 核心用户

#### 用户类型 A：轻量开发者

特征：

- 经常修改配置文件、脚本、文档、前端代码
- 不想为每次小修改都打开完整 IDE
- 对快捷键、Tab、查找、语言模式比较敏感

核心诉求：

- 打开快
- 工作区切换快
- 多文件切换顺手
- 不丢内容

#### 用户类型 B：普通桌面用户

特征：

- 主要编辑 txt、md、json、yaml、csv 等文件
- 更偏好中文界面
- 不熟悉复杂 IDE 信息架构

核心诉求：

- 一眼知道怎么开始
- 打开/保存路径明确
- 界面简洁，不吓人

#### 用户类型 C：持续写作者 / 文档维护者

特征：

- 会长时间编辑文本、Markdown、项目说明文档
- 对字体、主题、自动保存、会话恢复有强需求

核心诉求：

- 编辑舒适
- 崩溃或误关闭后可恢复
- 最近项目和最近文件能记住上下文

### 3.2 非目标用户

当前版本不面向以下人群作为主目标：

- 需要完整 IDE、调试器、插件生态的高级开发者
- 需要多人协作、云同步、AI 助手深度整合的团队用户
- 需要数据库管理、终端集成、Git 图形化操作全套能力的重度用户

## 4. 产品目标

### 4.1 业务目标

- 形成一个可公开分发的桌面编辑器产品基础版本
- 具备稳定安装、启动、编辑、保存、恢复的完整闭环
- 为后续命令面板、快速打开、全局搜索、Git 集成等能力提供产品与架构基础

### 4.2 用户目标

用户在第一次使用 30 秒内可以完成以下任意一个任务：

- 新建一个文件并开始输入
- 打开一个现有文件并编辑
- 打开一个文件夹作为工作区并浏览文件

### 4.3 质量目标

- 不再出现“空启动默认展示假文件树”的行为
- 新建文件后必须立即可编辑
- Tab 切换不能串内容
- 关闭应用时未保存内容必须有保护
- macOS Apple Silicon 设备可使用通用安装包正常安装

---

## 5. 产品原则

### 5.1 简洁优先

默认界面只展示最必要信息，不把高级能力强行堆到主界面。

### 5.2 桌面优先

优先使用桌面应用应有的交互方式，例如：

- 原生文件对话框
- 原生菜单和快捷键
- 最近项目
- 会话恢复
- 安装包与系统架构适配

### 5.3 可预测优先

用户的文件、内容和保存行为必须可理解、可恢复、可追踪。

### 5.4 逐层显露

对普通用户保持简单，对进阶用户提供逐层打开的能力入口。

---

## 6. 当前问题定义

基于当前产品现状，主要问题如下：

### 6.1 信息架构问题

- 在未打开文件/文件夹时默认出现资源管理器内容，破坏产品可信度
- “打开文件”和“打开文件夹”的产品入口不够清晰
- 空状态没有被认真设计

### 6.2 编辑模型问题

- 新建文件、Tab、文件路径、内容之间的绑定关系不稳定
- 编辑器实例与 Tab 状态未完全解耦
- 重命名、保存、另存为等流程不统一

### 6.3 UI 体验问题

- 工具栏、状态栏、标签栏样式不统一
- 语言模式入口视觉层级不足
- 字体选择太少，难以满足长期编辑体验

### 6.4 桌面分发问题

- macOS 打包未明确保证 Apple Silicon 兼容
- 发布流程没有形成稳定的跨平台构建策略

## 7. 版本规划

### 7.1 版本定义

| 版本 | 目标 | 说明 |
|------|------|------|
| V1 | 稳定可分发 | 建立“空启动 -> 打开 -> 编辑 -> 保存 -> 恢复”的完整闭环 |

### 7.2 本文档覆盖范围

本文档主要覆盖 V1，并为 V1.5 提供前置需求约束。

## 8. 核心用户任务

### 8.1 用户任务一：快速开始编辑

用户期望：

- 打开应用后马上知道下一步做什么
- 新建文件后即可输入
- 也可以不打开工作区，只编辑一个单独文件

成功标准：

- 30 秒内完成首次输入

### 8.2 用户任务二：管理一组文件

用户期望：

- 打开文件夹后看到真实、可操作的资源树
- 在多个标签间切换时不丢上下文
- 能新建、重命名、删除文件和文件夹

成功标准：

- 在工作区内编辑 3 个以上文件时，路径与内容关系始终正确

### 8.3 用户任务三：安全保存工作

用户期望：

- 保存行为明确
- 误关闭前有提醒
- 应用重启后可以恢复工作

成功标准：

- 退出时未保存内容不会静默丢失

### 8.4 用户任务四：调整编辑环境

用户期望：

- 能快速切换语言模式、主题、字体
- 设置有即时反馈
- 状态栏信息简洁但有用

成功标准：

- 常用设置在 2 步内完成调整

## 9. 核心场景设计

### 9.1 场景 A：空启动

前置条件：

- 用户刚启动应用
- 未打开任何文件或文件夹

系统表现：

- 展示欢迎态
- 提供三个主入口：
  - 新建文件
  - 打开文件
  - 打开文件夹
- 展示最近项目和最近文件（如有）
- 左侧资源管理器不展示伪造内容

验收标准：

- 未打开工作区时，资源管理器不显示任何默认目录树

### 9.2 场景 B：临时编辑单个文件

前置条件：

- 用户仅需打开并修改一个文件

系统表现：

- 支持系统文件选择器打开单文件
- 不要求先打开文件夹
- Tab 正确显示文件名
- 保存后原路径更新正确

验收标准：

- 单文件模式可独立运行，不依赖工作区

### 9.3 场景 C：打开文件夹作为工作区

前置条件：

- 用户选择一个目录

系统表现：

- 左侧显示真实文件树
- 中间支持多标签编辑
- 顶部显示工作区名
- 最近项目记录更新

验收标准：

- 工作区切换后文件树、当前选中文件、标签状态一致

### 9.4 场景 D：新建未保存文件

前置条件：

- 用户点击新建文件

系统表现：

- 立即创建未保存 Tab
- 默认聚焦编辑器
- Tab 支持改名
- 第一次保存触发另存为

验收标准：

- 新建文件后无需额外点击即可输入

### 9.5 场景 E：退出保护与恢复

前置条件：

- 存在未保存内容

系统表现：

- 关闭窗口或退出应用时弹出统一确认
- 支持：
  - 保存并退出
  - 不保存退出
  - 取消
- 异常退出后可恢复未保存会话

验收标准：

- 所有退出路径都统一经过保存保护策略

## 11. 信息架构

### 11.1 界面主区域

产品界面划分为五个区域：

1. 顶部主工具栏  
负责新建、打开、保存、切换侧栏、进入设置等全局动作

2. 左侧边栏  
在空工作区状态下展示引导；在工作区状态下展示文件树

3. 中间编辑区  
包含 Tab 区与 Monaco 编辑器

4. 右侧面板  
用于设置、未来的查找替换、命令面板结果等按需信息

5. 底部状态栏  
展示语言、主题、行列、编码、保存状态等轻量信息

### 11.2 模式定义

产品必须支持三种显式模式：

| 模式 | 说明 |
|------|------|
| `empty` | 未打开任何文件与工作区 |
| `single-file` | 仅打开一个或多个独立文件，不绑定工作区 |
| `workspace` | 已打开目录，文件树可用 |

验收标准：

- 不允许在 `empty` 模式下默认出现目录树

## 12. 交互需求

### 12.1 新建文件

交互规则：

- 点击后立即新建 `Untitled` Tab
- 焦点进入编辑器
- Tab 可双击重命名
- 第一次保存时使用另存为

### 12.2 打开文件

交互规则：

- 调用系统文件选择器
- 打开后进入 `single-file` 或当前工作区补充打开
- 不重复打开已存在 Tab

### 12.3 打开文件夹

交互规则：

- 调用系统目录选择器
- 成功后进入 `workspace` 模式
- 工作区名称立即可见
- 更新最近项目

### 12.4 保存

交互规则：

- 已有关联路径时直接保存
- 未命名文件必须触发另存为
- 保存成功后清除脏状态

### 12.5 重命名 Tab / 文件

交互规则：

- 未保存文件：只更新标签显示名与语言推断
- 已保存文件：执行真实重命名，并同步更新文件树与 Tab

### 12.6 关闭与退出

交互规则：

- 关闭单个脏 Tab 时需确认
- 关闭窗口或退出应用时统一弹出确认
- 支持保存全部

## 13. 视觉与体验需求

### 13.1 视觉方向

整体风格应符合：

- 轻专业
- 中文友好
- 层级清晰
- 桌面感强
- 避免廉价的网页套壳视觉

### 13.2 设计要求

- 空状态必须有明确文案和 CTA
- 工具栏、Tab、状态栏需采用统一设计语言
- 语言模式选择器需要有更强视觉辨识度
- 支持至少 3 套完整主题方向

### 13.3 字体要求

必须内置并暴露更丰富的字体选项，至少覆盖：

- JetBrains Mono
- Fira Code
- Maple Mono
- Cascadia Code
- IBM Plex Mono
- SF Mono / Menlo

---

## 14. 非功能需求

### 14.1 性能

- 冷启动目标：< 2 秒
- 单文件打开反馈：< 1 秒（常规文本）
- 工作区文件树首屏可见：< 1.5 秒（中小项目）

### 14.2 稳定性

- Tab 内容不能错位
- 文件路径不能丢失
- 保存失败必须可感知
- 文件系统异常必须有用户提示

### 14.3 兼容性

- macOS：支持 Intel 与 Apple Silicon
- Windows：主流 x64
- Linux：主流桌面发行版

### 14.4 可维护性

- 功能入口不应强绑定某个按钮
- 命令和快捷键应有统一注册中心
- 产品文档和架构文档需可持续增量维护

---

## 15. 成功指标

### 15.1 产品指标

- 首次启动后 30 秒内可成功进入编辑状态
- 首个公开版本具备稳定安装成功率
- 主要使用路径无阻塞：
  - 新建 -> 编辑 -> 保存
  - 打开文件 -> 编辑 -> 保存
  - 打开文件夹 -> 切换文件 -> 保存

### 15.2 质量指标

- 空启动下资源管理器错误展示率为 0
- 新建文件不可编辑问题为 0
- macOS M 系列无法打开包的问题为 0

## 16. 验收标准

### 16.1 P0 验收标准

- 启动后未打开文件夹时，不展示伪造目录树
- “打开文件”和“打开文件夹”均可通过系统对话框成功选择
- 新建文件后立即可编辑
- Tab 切换时内容、文件名、路径完全对应
- 未保存文件第一次保存时触发另存为
- 双击 Tab 可重命名
- 状态栏语言模式可正常切换
- macOS 发布流程支持 universal 包构建

### 16.2 P1 验收标准

- 最近项目和最近文件可持久化
- 会话恢复可恢复上次打开标签
- 自动保存可配置并可关闭
- 关闭脏标签与退出应用均有保护机制

## 18. 风险与依赖

### 18.1 风险

- Monaco 体积较大，可能影响启动性能
- 文件状态和编辑状态双写容易导致状态错乱
- 不同平台文件系统行为存在差异
- macOS 签名与公证会影响分发体验

### 18.2 依赖

- Tauri 文件系统与对话框能力稳定
- 前端状态模型重构完成
- 发布流程支持 universal target

## 20. 结论

这个编辑器当前最重要的不是继续扩展“功能列表”，而是先把它定义成一个可信、轻量、桌面感明确的产品。

V1 的任务只有一个：

把“从空白启动到稳定保存”的链路做成真正能反复使用的产品闭环。

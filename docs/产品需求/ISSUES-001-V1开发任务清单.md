# ISSUES-001: V1 开发任务清单

**文档版本：** 1.0  
**创建日期：** 2026-03-12  
**状态：** Draft  
**用途：** 用于拆分 GitHub Issues、Sprint Backlog、开发排期

---

## 1. 使用说明

本文档将 V1 工作拆成可直接转为 issue 的任务项。

字段说明：

- `ID`: 任务编号
- `里程碑`: 对应 `PLAN-001`
- `优先级`: P0 / P1 / P2
- `类型`: Feature / Refactor / Infra / UX / Docs / Test
- `说明`: 该任务要解决的问题
- `完成定义`: 什么状态算完成

---

## 2. M1 产品骨架稳定化

### TE-001 建立 workspaceStore

- 里程碑: M1
- 优先级: P0
- 类型: Refactor
- 说明: 新增 `workspaceStore`，统一管理 `empty` / `single-file` / `workspace` 模式、当前工作区路径和最近项目基础结构。
- 完成定义: UI 可通过 `workspaceStore.mode` 判断当前展示欢迎态、文件树态或单文件态。

### TE-002 空启动欢迎态重构

- 里程碑: M1
- 优先级: P0
- 类型: UX
- 说明: 设计并实现空启动欢迎态，提供“新建文件 / 打开文件 / 打开文件夹”三个主入口。
- 完成定义: 未打开任何文件或文件夹时，主界面显示明确欢迎态，不显示假文件树。

### TE-003 侧栏空状态改造

- 里程碑: M1
- 优先级: P0
- 类型: UX
- 说明: 资源管理器在无工作区时显示空状态说明，而不是展示伪造目录。
- 完成定义: 侧栏在无工作区时展示说明文案和打开文件夹入口。

### TE-004 打开文件流程统一

- 里程碑: M1
- 优先级: P0
- 类型: Feature
- 说明: 将“打开文件”统一接入系统文件选择器，并抽象为公共入口。
- 完成定义: Toolbar、欢迎态等入口均可复用同一打开文件流程。

### TE-005 打开文件夹流程统一

- 里程碑: M1
- 优先级: P0
- 类型: Feature
- 说明: 将“打开文件夹”统一接入系统目录选择器，并建立工作区切换逻辑。
- 完成定义: 选择目录后，正确进入 `workspace` 模式并刷新文件树。

### TE-006 最近项目与最近文件数据结构预埋

- 里程碑: M1
- 优先级: P1
- 类型: Refactor
- 说明: 在 store 层预留 `recentProjects` 和 `recentFiles` 数据结构，不要求首版完整 UI。
- 完成定义: 数据结构存在，可被打开文件/工作区流程写入。

## 3. M2 编辑模型重构

### TE-101 tabsStore 成为内容真源

- 里程碑: M2
- 优先级: P0
- 类型: Refactor
- 说明: 将每个标签的 `content`、`isDirty`、`language`、`filePath` 等状态集中到 `tabsStore`。
- 完成定义: 保存、切换、恢复时都从 `tabsStore` 读取内容。

### TE-102 editorStore 职责收缩

- 里程碑: M2
- 优先级: P0
- 类型: Refactor
- 说明: `editorStore` 仅保留活动编辑器 UI 态，如光标、撤销状态、只读状态。
- 完成定义: `editorStore` 不再作为多标签内容真源。

### TE-103 建立 tabService

- 里程碑: M2
- 优先级: P0
- 类型: Refactor
- 说明: 将标签创建、切换、关闭、关闭其他、关闭全部等业务逻辑下沉到 service 层。
- 完成定义: 组件不直接拼接复杂 tab 业务逻辑。

### TE-104 建立 fileService 统一文件业务入口

- 里程碑: M2
- 优先级: P0
- 类型: Refactor
- 说明: 将打开、保存、另存为、重命名、删除等文件业务逻辑统一到 `fileService`。
- 完成定义: UI 不直接操作底层文件命令。

### TE-105 新建未保存文件模型完善

- 里程碑: M2
- 优先级: P0
- 类型: Feature
- 说明: 支持 `isUntitled` 状态，确保新建文件立即可编辑，并在首次保存时触发另存为。
- 完成定义: 新建文件后可直接输入，首次保存行为可预测。

### TE-106 Tab 重命名规则统一

- 里程碑: M2
- 优先级: P0
- 类型: Feature
- 说明: 区分未保存文件和已保存文件的重命名策略。
- 完成定义: 未保存 Tab 重命名仅改展示名；已保存文件重命名同步真实路径。

### TE-107 单编辑器实例切换稳定化

- 里程碑: M2
- 优先级: P0
- 类型: Refactor
- 说明: 保持单 Monaco 实例，通过切换 value/model 响应不同标签。
- 完成定义: 多 Tab 切换时不串内容、不丢光标。

### TE-108 App.vue 业务逻辑下沉

- 里程碑: M2
- 优先级: P1
- 类型: Refactor
- 说明: 将 `App.vue` 中的业务流程逐步迁移到 services / commands。
- 完成定义: `App.vue` 主要负责布局与组装。

## 4. M3 恢复与保护闭环

### TE-201 建立 sessionService

- 里程碑: M3
- 优先级: P0
- 类型: Feature
- 说明: 新增 `sessionService`，负责会话快照的保存、恢复、清理。
- 完成定义: 会话快照结构稳定，可被应用启动和退出调用。

### TE-202 会话快照持久化

- 里程碑: M3
- 优先级: P0
- 类型: Feature
- 说明: 将当前工作区、Tab、活动标签、未保存内容、光标位置落盘。
- 完成定义: 应用重启后能恢复上次工作上下文。

### TE-203 未命名文件草稿恢复

- 里程碑: M3
- 优先级: P0
- 类型: Feature
- 说明: 未保存文件不写磁盘，但需要可恢复草稿。
- 完成定义: 崩溃或关闭后，未命名文件内容可恢复。

### TE-204 自动保存策略统一

- 里程碑: M3
- 优先级: P0
- 类型: Feature
- 说明: 已命名文件可自动保存，未命名文件转为会话保存。
- 完成定义: 自动保存开启时行为统一且可配置。

### TE-205 退出保护统一入口

- 里程碑: M3
- 优先级: P0
- 类型: Feature
- 说明: 新增统一关闭前检查，不允许多个组件各自处理退出确认。
- 完成定义: 关闭窗口或退出应用时，存在脏文件就弹统一确认。

### TE-206 windowService 建立

- 里程碑: M3
- 优先级: P1
- 类型: Refactor
- 说明: 抽象窗口关闭、窗口状态、未来原生菜单联动。
- 完成定义: 退出保护和会话保存不依赖具体组件生命周期。

## 9. 下一步建议

如果继续往下走，最推荐的实现顺序是：

1. `workspaceStore`

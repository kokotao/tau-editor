# 文本编辑器测试策略

## 1. 测试层次

```
┌─────────────────────────────────────┐
│           E2E 测试 (Playwright)      │  ← 用户场景验证
├─────────────────────────────────────┤
│         集成测试 (Vitest)           │  ← 模块间交互
├─────────────────────────────────────┤
│         单元测试 (Vitest)           │  ← 函数/组件验证
├─────────────────────────────────────┤
│         Rust 测试 (cargo test)      │  ← 后端逻辑验证
└─────────────────────────────────────┘
```

## 2. 单元测试策略

### 2.1 覆盖范围
- **工具函数**: 字符串处理、文件操作、编码转换
- **React 组件**: 编辑器组件、工具栏、状态栏
- **状态管理**: Store、Actions、Reducers
- **Hooks**: 自定义 Hooks 逻辑

### 2.2 测试框架
- **框架**: Vitest (与 Vite 集成)
- **断言库**: Vitest 内置 expect
- **Mock 库**: Vitest 内置 vi.mock
- **覆盖率**: 目标 > 80%

### 2.3 测试文件组织
```
tests/unit/
├── utils/
│   ├── string.test.ts
│   ├── file.test.ts
│   └── encoding.test.ts
├── components/
│   ├── Editor.test.tsx
│   ├── Toolbar.test.tsx
│   └── StatusBar.test.tsx
├── store/
│   ├── editorStore.test.ts
│   └── fileStore.test.ts
└── hooks/
    ├── useEditor.test.ts
    └── useFile.test.ts
```

## 4. E2E 测试策略

### 4.1 覆盖范围
- 完整用户工作流
- 跨平台兼容性
- 性能基准测试
- 回归测试

### 4.2 测试框架
- **框架**: Playwright
- **浏览器**: Chromium (Tauri 使用 Webview)
- **报告**: HTML + JUnit

### 4.3 测试场景
| 场景 | 优先级 | 描述 |
|------|--------|------|
| 新建并保存文件 | P0 | 创建新文件并保存到磁盘 |
| 打开现有文件 | P0 | 从磁盘打开文件并编辑 |
| Markdown 预览 | P1 | 切换预览模式验证渲染 |
| 查找替换 | P1 | 查找文本并替换 |
| 主题切换 | P2 | 切换亮暗主题验证持久化 |
| 多标签编辑 | P2 | 同时编辑多个文件 |
| 撤销重做 | P1 | 验证操作历史 |
| 导出功能 | P2 | 导出为 HTML/PDF |

### 4.4 测试文件组织
```
tests/e2e/
├── specs/
│   ├── file-operations.spec.ts
│   ├── editing.spec.ts
│   ├── markdown.spec.ts
│   └── settings.spec.ts
├── fixtures/
│   ├── test-files/
│   └── test-data/
└── utils/
    └── helpers.ts
```

## 6. CI/CD 测试流程

### 6.1 GitHub Actions 工作流
```yaml
# .github/workflows/test.yml
触发条件:
  - push: main, develop
  - pull_request: 所有分支

执行步骤:
  1. 代码检查 (lint)
  2. 类型检查 (tsc)
  3. 单元测试 (Vitest)
  4. Rust 测试 (cargo test)
  5. E2E 测试 (Playwright)
  6. 覆盖率报告
```

### 6.2 质量门禁
| 指标 | 要求 |
|------|------|
| 单元测试覆盖率 | ≥ 80% |
| E2E 测试通过率 | 100% |
| Lint 错误 | 0 |
| 类型错误 | 0 |

### 6.3 测试执行顺序
```
本地开发:
  1. 保存时运行 Lint + 类型检查
  2. Commit 前运行单元测试
  3. PR 前运行 E2E 测试

CI 环境:
  1. Lint → 类型检查 → 单元测试 → Rust 测试 → E2E 测试
```

## 9. 测试维护

### 9.1 更新策略
- 新功能必须附带测试
- Bug 修复必须添加回归测试
- 定期审查和清理过时测试

```typescript
// 格式：should_预期行为_当条件
describe('Editor', () => {

  it('should show error when file not found', () => {})
})
```

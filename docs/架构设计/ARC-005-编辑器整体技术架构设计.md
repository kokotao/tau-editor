# ARC-005: 编辑器整体技术架构设计

**文档版本：** 1.0  
**创建日期：** 2026-03-12  
**状态：** Draft  
**关联 PRD：** `PRD-001-轻量桌面文本编辑器正式产品需求文档.md`

---

## 1. 文档目标

本架构文档用于定义编辑器 V1 到 V1.5 阶段的整体技术结构，解决以下问题：

- 当前状态模型存在内容来源不统一的问题
- 单文件模式与工作区模式尚未抽象为清晰的产品模式
- 文件、Tab、编辑器实例、持久化恢复之间缺乏统一边界
- 发布流程与桌面能力尚未形成完整架构设计

本设计文档聚焦“稳定、可维护、可扩展”的桌面编辑器基础架构，而不是某个单点组件实现。

---

## 2. 架构目标

### 2.1 总体目标

- 支持 `empty` / `single-file` / `workspace` 三种产品模式
- 确保每个 Tab 的内容与文件身份一一对应
- 确保会话恢复、自动保存、退出保护有统一的数据基础
- 为命令面板、快速打开、查找替换、文件监听等后续能力预留结构

### 2.2 核心约束

- 内容的单一事实来源必须明确
- 编辑器实例数量应最小化
- 文件系统调用必须经过服务层
- 桌面原生能力要通过清晰边界对前端暴露

---

## 3. 当前问题复盘

### 3.1 状态双写问题

当前实现中，`editorStore` 与 `tabsStore` 都可能参与内容状态维护，容易导致：

- Tab 切换后内容错位
- 保存时读取到非当前标签内容
- 新建文件与旧文件状态污染

### 3.2 模式边界不清

产品当前未将以下概念明确分开：

- 空启动
- 单文件编辑
- 工作区编辑

导致：

- 资源管理器显示逻辑错误
- 欢迎态难以设计
- 最近项目和最近文件难以抽象

### 3.3 桌面能力分散

当前“打开文件”“打开文件夹”“保存”“重命名”等行为分散在组件与 store 之间，后续将难以复用到：

- 原生菜单
- 快捷键
- 命令面板
- 自动化测试

## 5. 分层设计

## 5.1 UI 层

职责：

- 展示状态
- 分发用户交互
- 不直接调用底层文件系统

UI 层应只依赖：

- store 的响应式状态
- command handler / service 暴露的动作

UI 层不应承担：

- 路径合法性判断
- 保存策略决策
- 会话恢复逻辑

## 5.2 Command 层

职责：

- 统一注册所有可执行动作
- 给不同入口提供同一套执行逻辑

入口来源包括：

- Toolbar 按钮
- 快捷键
- 原生菜单
- 命令面板
- 欢迎页 CTA

建议结构：

```typescript
interface AppCommand {
  id: string;
  title: string;
  category: 'file' | 'edit' | 'view' | 'workspace' | 'help';
  shortcut?: string;
  when?: (context: CommandContext) => boolean;
  run: (context: CommandContext) => Promise<void> | void;
}
```

设计价值：

- 后续新增命令面板无需重写业务逻辑
- UI 只是 command 的展示入口

## 5.3 Service 层

职责：

- 编排业务流程
- 调用 store
- 调用 Tauri bridge
- 处理错误与反馈

建议服务拆分如下：

### workspaceService

负责：

- 打开工作区
- 切换工作区
- 关闭工作区
- 刷新文件树
- 维护最近项目

### fileService

负责：

- 打开文件
- 保存文件
- 另存为
- 新建文件
- 重命名文件
- 删除文件

### tabService

负责：

- 创建标签
- 切换标签
- 关闭标签
- 关闭其他/全部
- 标签重命名

### sessionService

负责：

- 保存会话快照
- 恢复上次标签
- 恢复未保存草稿
- 清理过期会话数据

### dialogService

负责：

- 打开文件选择器
- 打开文件夹选择器
- 另存为对话框
- 退出确认对话框

### windowService

负责：

- 应用关闭前钩子
- 窗口状态持久化
- 后续原生菜单联动

## 6. 状态模型设计

## 6.1 设计原则

- 内容真源在 `tabsStore`
- `editorStore` 只维护当前活动编辑器 UI 态
- `workspaceStore` 负责产品模式、工作区、最近项目
- 所有持久化状态集中管理，不散落到组件

## 6.2 workspaceStore

建议新增或重构为：

```typescript
type AppMode = 'empty' | 'single-file' | 'workspace';

interface RecentProject {
  path: string;
  name: string;
  openedAt: number;
}

interface RecentFile {
  path: string;
  name: string;
  openedAt: number;
}

interface WorkspaceState {
  mode: AppMode;
  currentWorkspacePath: string | null;
  currentWorkspaceName: string | null;
  recentProjects: RecentProject[];
  recentFiles: RecentFile[];
  isReady: boolean;
}
```

职责：

- 定义当前产品模式
- 管理最近项目和最近文件
- 决定左侧边栏显示欢迎态还是文件树

## 6.3 tabsStore

建议以标签为内容真源：

```typescript
interface TabModel {
  id: string;
  filePath: string | null;
  fileName: string;
  language: string;
  encoding: string;
  content: string;
  isDirty: boolean;
  isUntitled: boolean;
  isPinned: boolean;
  cursor: { line: number; column: number };
  selection: { start: number; end: number } | null;
  openedAt: number;
  lastActivatedAt: number;
}

interface TabsState {
  tabs: TabModel[];
  activeTabId: string | null;
  recentClosedTabs: string[];
}
```

关键约束：

- 所有标签内容必须保存在 `tabsStore`
- 切换标签时，只是切换活动 `tabId`
- 保存和恢复均从 `tabsStore` 取数据

## 6.4 editorStore

建议收敛为活动编辑器的 UI 状态：

```typescript
interface EditorUiState {
  activeLanguage: string;
  activeEncoding: string;
  isReadOnly: boolean;
  canUndo: boolean;
  canRedo: boolean;
  cursorPosition: { line: number; column: number };
  selection: { start: number; end: number } | null;
  lineCount: number;
}
```

不再承担：

- 全局内容真源
- 多标签内容缓存

## 6.5 settingsStore

维持并扩展：

- 主题
- 字体
- 自动保存
- 文件树宽度
- 恢复会话开关
- 忽略隐藏文件

## 6.6 notificationStore

维持现有能力，但建议增加：

- `persistent`
- `dismissAllByScope`
- `progress`

以支持后续大文件加载、工作区扫描等反馈

## 7. 编辑器实例设计

## 7.1 单编辑器实例原则

整个主界面只维护一个 Monaco 编辑器实例。

原因：

- 多实例会造成性能浪费
- 与多标签数据同步更复杂
- 绝大多数场景不需要多个同时可见编辑器

## 7.2 切换机制

切换 Tab 时流程如下：

1. 从 `tabsStore` 读取目标标签
2. 将其 `content / language / cursor / selection` 同步给 `EditorCore`
3. `EditorCore` 更新 model 或 value
4. `editorStore` 更新活动 UI 状态

## 7.3 编辑过程

输入时流程如下：

1. `EditorCore` 触发 `content-change`
2. 通过 `tabService` / `tabsStore` 更新当前活动标签内容
3. `editorStore` 更新行列、撤销状态等 UI 信息
4. 自动保存调度器决定是否进入后台保存

## 8. 文件系统架构设计

## 8.1 Tauri Bridge 边界

前端不应散落直接调用 `invoke`，建议集中在 bridge 层统一封装：

```typescript
fileCommands.readFile(path)
fileCommands.writeFile(path, content)
fileCommands.listFiles(dir)
fileCommands.renameFile(oldPath, newPath)
fileCommands.deleteFile(path)
fileCommands.getFileInfo(path)
```

扩展建议：

```typescript
workspaceCommands.watchWorkspace(path)
workspaceCommands.unwatchWorkspace(path)
sessionCommands.saveSession(snapshot)
sessionCommands.loadSession()
```

## 8.2 文件树数据结构

```typescript
interface FileTreeNode {
  name: string;
  path: string;
  type: 'file' | 'folder';
  isExpanded?: boolean;
  children?: FileTreeNode[];
  isLoaded?: boolean;
  gitStatus?: 'modified' | 'added' | 'deleted' | 'untracked';
}
```

设计要求：

- 支持懒加载
- 支持刷新局部节点
- 支持未来的文件监听局部更新

## 8.3 文件监听

V1.5 建议引入文件监听服务：

职责：

- 检测外部修改
- 检测外部删除
- 反馈给当前打开 Tab

用户策略：

- 文件外部变更时提示“重新加载 / 忽略”
- 如果当前文件有本地未保存修改，必须给冲突提示

## 9. 会话恢复架构

## 9.1 恢复目标

支持以下恢复对象：

- 上次打开的工作区
- 上次打开的标签列表
- 当前激活标签
- 未保存内容草稿
- 光标位置

## 9.2 会话快照结构

```typescript
interface SessionSnapshot {
  version: number;
  savedAt: number;
  mode: 'empty' | 'single-file' | 'workspace';
  workspacePath: string | null;
  activeTabId: string | null;
  tabs: Array<{
    id: string;
    filePath: string | null;
    fileName: string;
    language: string;
    encoding: string;
    content: string;
    isDirty: boolean;
    isUntitled: boolean;
    cursor: { line: number; column: number };
  }>;
}
```

## 9.3 保存策略

- 标签变更后节流保存会话快照
- 应用切后台 / 窗口关闭前强制落盘
- 自动保存文件与会话保存分离

## 9.4 恢复策略

恢复优先级：

1. 读取本地会话快照
2. 恢复工作区路径
3. 恢复标签
4. 已保存文件重新读取磁盘
5. 未保存文件优先恢复草稿内容

异常处理：

- 文件不存在：显示“文件缺失”状态标签
- 读取失败：标签保留但提示错误

## 10. 保存与自动保存策略

## 10.1 保存分类

### 显式保存

由用户触发：

- Toolbar 保存
- 菜单保存
- 快捷键保存
- 命令面板保存

### 自动保存

由系统触发：

- 达到配置时间间隔
- 切换窗口时（可选）
- 失焦时（后续可选）

## 10.2 自动保存约束

- 未命名文件不直接自动落盘，只保存为会话草稿
- 已命名文件可自动写盘
- 自动保存失败必须进入通知中心

## 10.3 退出保护

关闭应用前由 `windowService` 统一判断：

```typescript
if (tabsStore.hasDirtyTabs()) {
  showConfirmDialog();
}
```

严禁在多个组件中各自实现退出确认逻辑。

## 12. 组件职责建议

## 12.1 App.vue

职责：

- 编排主布局
- 组装主要 store 和 service
- 不直接承担复杂业务逻辑

应逐步减少 App.vue 中的业务实现，把逻辑下沉到 service / command。

## 12.2 Toolbar.vue

职责：

- 只负责展示和触发 command

## 12.3 FileTree.vue

职责：

- 展示树
- 抛出节点交互事件
- 不直接操作文件系统

## 12.4 EditorTabs.vue

职责：

- Tab 展示
- Tab 交互事件
- 不持有编辑器实例

## 12.5 EditorCore.vue

职责：

- 单一 Monaco 实例封装
- 暴露最小编辑接口
- 响应 value / language / readOnly 切换

## 15. 错误处理设计

## 15.1 错误分层

### 用户可理解错误

例如：

- 文件不存在
- 没有写权限
- 保存失败

处理方式：

- 通知提示
- 提供重试入口

### 系统级错误

例如：

- Tauri invoke 失败
- 会话恢复数据损坏

处理方式：

- 记录日志
- 提供降级逻辑

## 15.2 降级策略

- 会话恢复失败时退回欢迎页
- 文件树加载失败时不阻塞编辑器主体
- 单个文件读取失败不影响其他标签

## 17. 风险与决策记录

### 风险 1：继续让 editorStore 保存全局内容

结果：

- 高概率继续引发 Tab 内容串联问题

决策：

- 不采用，内容真源放在 `tabsStore`

### 风险 2：每个 Tab 一个 Monaco 实例

结果：

- 性能差
- 状态同步复杂

决策：

- 不采用，维持单编辑器实例

### 风险 3：命令逻辑继续分散在 UI

结果：

- 后续命令面板、菜单、快捷键难以统一

决策：

- 引入 command registry

## 19. 结论

本项目后续最重要的技术工作不是继续零散加功能，而是完成三件事：

1. 统一产品模式抽象  
2. 统一内容真源  
3. 统一命令与桌面能力入口

完成这三件事之后，命令面板、快速打开、查找替换、Git 状态、自动更新等能力都能自然往上长，而不会继续让主应用变成难以维护的大组件。

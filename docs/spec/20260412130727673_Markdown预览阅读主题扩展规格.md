# Markdown 预览阅读主题扩展规格

**文档版本：** 1.0  
**创建时间：** 2026-04-12 13:07  
**状态：** Approved  
**目标范围：** `frontend/src` Markdown 预览主题能力扩展

---

## 1. 背景

当前项目中的 Markdown 预览区只有“跟随整体明暗主题”的基础能力，缺少面向阅读场景的独立预览主题。现有实现存在以下问题：

- Markdown 预览不能单独切换阅读风格
- 预览区样式主要依赖全局面板变量，缺少独立的内容阅读 tokens
- 长文、文档、博客、夜间阅读等场景无法快速切换到更合适的视觉方案
- 设置面板和预览区右键菜单都没有提供预览主题入口

本次扩展目标是在不影响编辑器 Monaco 主题和全局应用主题体系的前提下，为 Markdown 预览新增独立阅读主题。

---

## 2. 设计目标

- 为 Markdown 预览新增独立的主题设置项，并持久化保存
- 在设置面板与预览区右键菜单中都提供主题切换入口
- 首批提供 4 套主题，覆盖文档站、护眼、出版感、夜读四类场景
- 每套主题不只切换背景色，还应统一控制标题、正文、引用、表格、代码块、链接、菜单等视觉元素
- 改动范围保持在前端层，不修改 Tauri/Rust 后端逻辑

---

## 3. 非目标

- 不改 Monaco 编辑器主题逻辑
- 不改全局 `themeSkin` 与 `customThemeColors` 体系
- 不新增工具栏主题入口
- 不支持用户自定义 Markdown 预览主题颜色
- 不引入新的第三方主题库

---

## 4. 用户确认结论

本次需求的已确认设计结论如下：

- 主题模式：Markdown 预览主题独立切换
- 主题风格：混合型主题组合
- 切换入口：设置面板 + 预览区右键菜单
- 实现策略：平衡方案

首批落地主题如下：

1. `docs-clean` / 文档站清朗
2. `paper-soft` / 护眼纸张
3. `editorial-warm` / 出版暖页
4. `graphite-night` / 夜读石墨

---

## 5. 信息架构与状态设计

### 5.1 新增设置项

在 `frontend/src/stores/settings.ts` 的 `EditorSettings` 中新增：

```ts
markdownPreviewTheme: 'docs-clean' | 'paper-soft' | 'editorial-warm' | 'graphite-night';
```

默认值建议：

```ts
markdownPreviewTheme: 'docs-clean'
```

原因：

- 作为默认主题，`docs-clean` 对大部分 Markdown 场景最稳妥
- 视觉中性、信息密度均衡，更适合作为首次使用体验

### 5.2 持久化要求

以下流程都必须纳入现有 `localStorage` 设置持久化：

- `loadFromStorage()`
- `saveToStorage()`
- `updateSettings()`
- `resetToDefaults()`

在 [SettingsPanel.vue](/Users/albertluo/workSpace/albertLuo/newThing/text-editor/frontend/src/components/editor/SettingsPanel.vue) 的编辑器设置区域新增一个选择项：

放置位置建议：

### 6.2 预览区右键菜单

在 [MarkdownPreview.vue](/Users/albertluo/workSpace/albertLuo/newThing/text-editor/frontend/src/components/editor/MarkdownPreview.vue) 的 `buildMenuItems()` 中新增一组菜单项：

要求：

- 点击后立即生效

## 7. 视觉设计要求

### 7.1 主题应用范围

主题必须覆盖以下元素：

- 预览容器背景
- 正文文本颜色
- 标题 `h1` `h2` `h3`
- 行内代码
- 代码块 `pre code`
- 引用块 `blockquote`
- 分割线 `hr`
- 链接 `a`
- 表格 `table` `th` `td`
- Mermaid 错误提示
- 右键菜单面板与 hover 态

### 7.2 主题定义建议

建议使用“主题类名 + CSS 自定义属性”的方式，不将大量判断散落到模板中。

建议结构：

- 预览根节点根据当前主题追加类名
- 每套主题定义一组 preview tokens
- 组件样式统一读取这些 tokens

示例方向：

```ts
markdown-preview--docs-clean
markdown-preview--paper-soft
markdown-preview--editorial-warm
markdown-preview--graphite-night
```

### 7.3 各主题定位

#### 文档站清朗

- 白底或偏冷白背景
- 蓝灰标题和中性正文
- 清爽、专业、产品文档站风格

#### 护眼纸张

- 暖米色背景
- 柔和棕灰文字
- 适合长时间阅读与知识笔记

#### 出版暖页

- 更强排版层次
- 标题、引用、代码块更有质感
- 适合文章、博客、长篇 Markdown

#### 夜读石墨

- 深石墨底
- 低眩光、高可读性
- 适合夜间阅读和暗色工作流

## 9. 测试与验收标准

### 9.1 功能验收

- 设置面板可切换 Markdown 预览主题
- 右键菜单可切换 Markdown 预览主题
- 主题切换立即反映到预览区
- 刷新页面后仍能恢复上次选择的主题
- 切换主题不会影响 Monaco 编辑器主题

### 9.2 UI 验收

- 4 套主题在标题、正文、代码块、引用、表格上都有明显区分
- 浅色主题下代码块仍具可读性
- 暗色主题下正文、链接、提示状态不过曝
- 右键菜单颜色能跟随当前 Markdown 预览主题或保持视觉协调

### 9.3 回归关注点

- Markdown 预览的已有右键菜单能力不能退化
- Mermaid 渲染与错误提示不能被主题样式破坏
- 滚动同步功能不能受新增类名影响
- `SettingsPanel` 的现有选择器和动画不能异常

## 10. 风险与规避

### 风险 1：样式逻辑继续堆在组件里

风险：

- `MarkdownPreview.vue` 样式继续膨胀，后续扩展困难

规避：

- 使用统一 preview tokens
- 必要时抽离 `markdownPreviewTheme.ts`

### 风险 2：用户混淆 Monaco 主题与 Markdown 预览主题

风险：

- 用户误以为切换预览主题会影响编辑器配色

规避：

- 设置项文案明确写为“Markdown 预览主题”
- 与 “编辑器主题” 分开展示

### 风险 3：右键菜单项过长

风险：

- 菜单过高或语义不清

规避：

- 使用统一前缀或分组
- 当前主题项禁用显示，降低误操作

---

## 11. 最终结论

本次需求采用“独立预览主题 + 设置面板/右键菜单双入口 + 4 套首发主题”的实现方式。

这是一个中等规模但边界清晰的前端增强需求，建议先按以下顺序实施：

1. 增加设置状态与文案
2. 增加设置面板切换入口
3. 增加预览组件主题样式和右键菜单切换
4. 补齐单测与构建验证
